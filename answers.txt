part 1
Q1. When I type google.com into my browser and press Enter, several things happen behind the scenes in just a few seconds.

First, my browser asks the DNS (Domain Name System) to find the IP address of google.com because computers communicate using numbers, not website names.

Once the IP address is found, my request travels across the internet to Google's servers. Before the request reaches the website, it usually passes through a firewall, which acts like a security guard by checking that the request is safe and blocking anything suspicious.

Next, the request may go through a load balancer, which works like a traffic conductor. If many people are visiting Google at the same time, the load balancer decides which server should handle my request so that no single server becomes overloaded.

The request then reaches the web server, which receives it. If the page requires processing, the web server passes the request to the application server. The application server contains the website's business logic and decides what needs to happen next. If information needs to be retrieved, such as search results or account details, it communicates with the database.

The database sends the requested information back to the application server, which processes it and sends it to the web server. Finally, the web server returns the completed webpage to my browser, and the page appears on my screen.

Although all these steps happen in the background, they usually take only a fraction of a second.
A client is a device or application that requests information or services from another computer over the internet.
A server is a computer that receives those requests, processes them, and sends back the requested information.
For example:
•	If I open YouTube on my laptop, my laptop and browser are the client because they are requesting videos. 
•	Google's YouTube servers are the servers because they store the videos and send them back to my browser. 
In summary, the client asks, while the server answers.
Q3. DNS (Domain Name System) is like the internet's phonebook.

People prefer remembering names like google.com instead of long numbers called IP addresses. DNS translates a website name into its IP address so that computers know where to find the website.

Without DNS, we would have to remember the IP address of every website we wanted to visit, which would be very difficult.
Q4. A firewall protects a server by checking incoming and outgoing network traffic. It blocks harmful or unauthorized requests while allowing safe ones through. It is mainly responsible for security.
A load balancer distributes incoming requests across multiple servers so that no single server becomes too busy. This helps websites stay fast and available even when many people are using them at the same time.
The main difference is that a firewall focuses on protecting the system, while a load balancer focuses on sharing work efficiently among servers to avoid system crash. 


Q5. A web server is responsible for receiving requests from users and delivering static content such as HTML pages, CSS files, JavaScript files, and images. If a request requires processing, such as logging in or retrieving user information, the web server passes it to the application server.
An application server is responsible for handling the backend logic of a website. It processes user requests, runs the backend code, communicates with the database, and sends the appropriate response back to the web server, which then returns it to the client.
Backend developers work more with the application server because it is where they write and manage the code that powers the website's functionality. For example, when a user logs in, registers an account, makes a payment, or retrieves information from a database, the application server is responsible for carrying out these tasks.
In simple terms: the web server is like a receptionist who receives visitors and directs them where to go, while the application server is like the office staff who do the actual work behind the scenes.

Q6. The client never communicates directly with the database because it would create serious security risks. If users could access the database directly, they might accidentally or intentionally view, change, or delete important information without permission.
Instead, the client sends its request to the application server, which acts as a middleman between the client and the database. The application server first checks whether the request is valid and whether the user is authorized to perform the requested action. Only then does it communicate with the database.
Having an application server between the client and the database has several benefits:
•	It keeps the database secure from unauthorized access. 
•	It checks whether users have permission to perform certain actions, such as viewing or editing information. 
•	It validates user input to reduce errors and prevent harmful data from being stored. 
•	It applies the application's business rules before interacting with the database. 
•	It prevents users from accessing sensitive information directly. 
For example, imagine you want to check your MTN MoMo account balance. You don't connect directly to MTN's database. Instead, you log in through the MoMo app. The app sends your request to the application server, which first verifies your username and password. If your login details are correct, the application server retrieves your account balance from the database and sends it back to your phone. This process ensures that only you can access your account information and that the database remains protected.
In simple terms: think of the application server as a bank teller. You don't walk into the bank's vault to collect your money. Instead, you ask the teller, who verifies your identity, retrieves the information or money from the vault, and safely gives it to you. In the same way, the application server safely handles communication between the client and the database.

Q7. A university department page, such as ug.edu.gh/statistics, is more likely to be a static website because most of the information it displays is the same for every visitor. Details such as academic programmes, staff information, contact details, and department news do not change based on who is viewing the page. Although the content may be updated occasionally by the university, every visitor sees the same information.
On the other hand, an MTN MoMo account page is a dynamic website because the content changes depending on the user who logs in. For example, when you sign in, you see your own account balance, transaction history, and personal details, while another user will see completely different information. The page retrieves this information from a database and displays it specifically for the logged-in user.
In simple terms: a static website is like a notice board on campus that everyone reads and sees the same information. A dynamic website is like your student portal or MoMo account, where the information is personalized and unique to you.

Q8. The main difference between frontend and backend development is that the frontend is what users can see and interact with, while the backend is what happens behind the scenes to make the application work.
The frontend includes things like the website's layout, colours, buttons, forms, and menus. It focuses on creating a good user experience. The backend, however, handles tasks such as processing requests, managing databases, storing and retrieving information, and applying the application's business logic.
If I were building the login feature of a website, the backend would be responsible for checking the user's password against the database. This is because passwords are sensitive information and should never be verified on the frontend, where users can inspect or manipulate the code. The backend securely compares the entered password with the encrypted password stored in the database and then tells the frontend whether the login was successful or not.
For example, when you log into your University of Ghana MIS Web or the MTN MoMo app, you only enter your username and password on the login page (the frontend). The backend then securely checks your credentials in the database. If they are correct, it grants you access to your account. If they are incorrect, it returns an error message such as "Invalid username or password."
In simple terms: the frontend is like the receptionist who welcomes you and collects your information, while the backend is like the office staff who verify your details and decide whether you should be allowed access. The receptionist doesn't make the final decision—the office staff do.
